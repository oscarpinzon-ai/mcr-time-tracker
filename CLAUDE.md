## Project Overview

**MCR Tech Performance Tool** is a technician time-tracking app for **Modern Compactor Repair** (Austin, TX). Technicians clock in/out of jobs assigned via HouseCall Pro (HCP). Admins see live crew status, edit time entries, and export reports.

**Phase 1 (DONE — Lovable):** Frontend skeleton, Supabase tables, branding, CRUD, technician flow, admin dashboard with 4 tabs.

**Phase 2 (YOU — Claude Code):** HouseCall Pro API integration, auto clock-out, PDF/Excel exports, employee sync. This document is your spec.

## Database Schema (already migrated)

**employees** — id (uuid PK), name, hcp_employee_id, is_active, role ('technician'|'admin'), created_at
**time_entries** — id (uuid PK), employee_id (FK→employees), hcp_job_id, job_number, customer_name, job_type, job_address, clock_in, clock_out, status ('active'|'paused'|'completed'), total_minutes, created_at
**pause_logs** — id (uuid PK), time_entry_id (FK→time_entries CASCADE), pause_start, pause_end
**admin_edits_log** — id (uuid PK), time_entry_id (FK→time_entries CASCADE), edited_by, field_changed, old_value, new_value, edited_at
**hcp_jobs_cache** — id (uuid PK), hcp_job_id (UNIQUE), job_number, customer_name, job_type, job_address, status, scheduled_date, assigned_employee_ids (text[]), last_synced_at, raw_data (jsonb)

RLS is enabled on all tables with open "Allow all" policies (to be tightened in a future phase).

## Phase 2 Tasks

### Task 1: HouseCall Pro API Proxy (Server-Side)

**Goal:** Create a server-side API route (TanStack Start server function or Cloudflare Worker route) that proxies requests to HouseCall Pro's REST API. The frontend must NEVER call HCP directly.

**HCP API Docs:** https://docs.housecallpro.com/

**Authentication:** HCP uses OAuth2 Bearer token. Store the API key as environment variable `HCP_API_KEY` (Cloudflare Worker secret / `.dev.vars`).

**Endpoints to proxy:**

1. `GET /jobs` — Fetch jobs with filters (scheduled date range, status).
   - HCP endpoint: `GET https://api.housecallpro.com/pro/v1/jobs`
   - Query params: `scheduled_start_min`, `scheduled_start_max`, `work_status[]`
   - Map HCP response to our `hcp_jobs_cache` schema

2. `GET /employees` — Fetch all employees/pros.
   - HCP endpoint: `GET https://api.housecallpro.com/pro/v1/employees`
   - Map to our `employees` table (match by name or hcp_employee_id)

3. `GET /schedule` — Fetch schedule entries to determine which employee is assigned to which job.
   - HCP endpoint: `GET https://api.housecallpro.com/pro/v1/jobs/{job_id}` (includes `assigned_employees`)
   - Or use the jobs endpoint which returns `dispatched_employees` array

**Implementation approach:**
- Create server functions in `src/routes/api/` directory using TanStack Start's `createAPIFileRoute` or `createServerFn`
- Example: `src/routes/api/hcp-sync.ts` — a POST endpoint that fetches from HCP and upserts into `hcp_jobs_cache`
- Use the server-side Supabase client (`src/integrations/supabase/client.server.ts`) to write to the database
- Handle pagination (HCP returns max 200 per page)

**Data mapping from HCP to our schema:**

```
HCP Job → hcp_jobs_cache:
  job.id → hcp_job_id
  job.job_number → job_number  
  job.customer.name (or first+last) → customer_name
  job.work_line_items[0].name or job.tags → job_type (map to: 'Service Call / Repair', 'Yard Work', 'Installation / Removal')
  job.address.street + city + state + zip → job_address
  job.work_status → status (map: 'scheduled'→'scheduled', 'in_progress'→'in_progress', 'complete'→'completed', etc.)
  job.schedule.scheduled_start (date portion) → scheduled_date
  job.dispatched_employees[].id → assigned_employee_ids
  full response → raw_data
```

**Note:** The exact HCP response schema may differ — inspect the actual API response and adjust mapping accordingly. Store `raw_data` as jsonb for debugging.

### Task 2: Polling / Scheduled Sync

**Goal:** Automatically sync jobs from HCP every 3 minutes during business hours.

**Schedule:** Every 3 minutes, 6:00 AM – 7:00 PM CDT, Monday through Saturday.

**Implementation options (pick the best for this stack):**

**Option A — Cloudflare Cron Trigger (preferred):**
Add a cron trigger in `wrangler.jsonc`:
```jsonc
"triggers": {
  "crons": ["*/3 6-19 * * 1-6"]  // Every 3 min, 6AM-7PM, Mon-Sat (UTC — adjust for CDT)
}
```
The cron handler calls the same sync logic as Task 1.

**Option B — Supabase pg_cron:**
Create a Supabase database function + pg_cron job that calls an Edge Function.

**Option C — Client-side polling (fallback):**
If neither server-side option works cleanly within the Lovable/Cloudflare setup, implement client-side polling in the technician view only:
- `setInterval` every 3 minutes when the technician page is open
- Call a server function that does the HCP fetch + upsert
- Only active during 6AM-7PM CDT Mon-Sat

**Sync logic:**
1. Fetch jobs from HCP where `scheduled_start` is today (CDT)
2. Upsert into `hcp_jobs_cache` (ON CONFLICT hcp_job_id DO UPDATE)
3. Log `last_synced_at` timestamp
4. For each job, extract `dispatched_employees` → `assigned_employee_ids`

**CDT timezone:** Use `America/Chicago`. All time comparisons for "today" and business hours must use this timezone.

### Task 4: Auto Clock-Out at 7:00 PM CDT

**Goal:** Any `time_entry` with `status='active'` or `status='paused'` at 7:00 PM CDT gets automatically clocked out.

**Implementation:**
- Add a Cloudflare Cron Trigger or Supabase pg_cron that runs daily at 7:00 PM CDT (Mon-Sat)
- Query: `SELECT * FROM time_entries WHERE status IN ('active', 'paused') AND clock_out IS NULL`
- For each entry:
  1. Close any open `pause_logs` (set `pause_end = 7:00 PM CDT`)
  2. Set `clock_out = 7:00 PM CDT` on the `time_entry`
  3. Calculate `total_minutes` (clock_out - clock_in - total pause time)
  4. Set `status = 'completed'`
  5. Insert into `admin_edits_log`: `edited_by='System (Auto Clock-Out)', field_changed='clock_out', old_value=NULL, new_value='7:00 PM CDT timestamp'`
